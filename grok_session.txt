### 3. Go into More Depth on the Borrow Checker (This Session's Core Addition)
**User request:** "Go into more depth on the borrow checker, how it works, best practices..."
- This was listed as planned in the summary: "References (`&`, `&mut`), Ownership, and Borrowing".
- Response: Created dedicated `move/references.md` (in-depth file) + updated `move_summary.md` (TOC, core concepts table, basic topics list) to integrate it live. Also added cross-refs.
- Content sources: Direct from move-book.com (references.md, ownership-and-scope.md, reference/primitive-types/references.md), docs.sui.io PTB page (borrowing rules), cross-referenced with abilities/objects.

**Full depth added (summary of the new file's content — for this log and future pickup):**

**Ownership Basics:**
- Every value has one owner (scope/block/function). End of scope → drop (unless `drop` ability).
- Move semantics on pass-by-value / return: transfers ownership. Original var invalid after.
- Sub-scopes via `{}` blocks.
- Copyable types (copy ability: u64 etc. + some structs) auto-copied in many places. Use `move` keyword to force transfer.
- Example code for owner/move/return/blocks.

**References (Borrowing):**
- `&T` (immut/read) and `&mut T` (mut/read+write).
- "Lend" without losing ownership. Borrower gets temporary view.
- Examples: metro card `is_valid(&card)`, `enter(&mut card)`, `recycle(card)` (by-value consume).
- Operators: `&e`, `&mut e`, `&e.f` / `&mut e.f` (field borrows, extendable), `freeze(e)` (mut→immut).
- No `&&T` (references to references forbidden).

**How the Borrow Checker Works:**
- Static (compile-time) + on-chain bytecode verifier.
- Tracks: ownership, active borrows (refs to values/fields/paths), mutability.
- **Core rules** (enforced in pure Move + PTB runtime):
  - Mutable borrow: No other borrows (mut or immut) outstanding. (Prevents aliasing writes / data races.)
  - Immutable borrow: No outstanding mutable borrows. Multiple immut OK.
  - Move (by value): No borrows outstanding (would invalidate refs).
  - Copy: Always OK (even with borrows).
- Freeze inference + subtyping: `&mut T` <: `&T` (compiler auto-freezes where a `&T` is expected; appears in errors).
- Read (`*ref`): Requires `T: copy` (can't copy resources via ref).
- Write (`*ref = v`): Requires `T: drop` (write discards old value).
- Refs are **ephemeral only** — cannot be stored in structs/objects (unlike Rust). Destroyed at tx end.
- Field borrows & reborrows supported (within module for deep nesting).
- PTB specifics: Runtime enforces same rules on input/result "array". Move calls can't return refs. Objects have special handling (Receiving<T>, shared object restrictions). TxContext injected specially.

**Common Errors the Checker Catches (with code examples in the file):**
- Use after move.
- Conflicting borrows (two &mut, & while &mut active).
- Move while borrowed.
- Write without drop (destroys resource).
- Attempt to store a reference.
- In PTBs: cross-command borrow/move violations.

**Sui-Specifics:**
- Objects (key ability) + refs: `&mut MyObj` in entry fns for mutation of shared/owned.
- Capabilities: Pass `&Cap` / `&mut Cap` (not by value) for access control.
- Owned objects often moved into transfers; shared use &mut.
- UID handling: Create in &mut TxContext, careful with delete.
- Additional Sui verifier rules on top of borrow checker (e.g., shared objects must remain shared or be deleted by end of tx).

**Best Practices (12 listed in file):**
1. Prefer & / &mut over move for temp access (keeps ownership, great for caps/objects).
2. & for reads, &mut only for writes.
3. `let mut` for locals you'll &mut-borrow.
4. Tight borrow scopes.
5. Sui objects: entry fns → `&mut Obj`; owned often by-value for consume/transfer.
6. Capabilities: & / &mut Cap (never transfer unless intended).
7. Don't fight checker — question if you *really* need ownership.
8. Granular field reborrows.
9. Tests/PTBs surface same errors.
10. Read full error messages (very precise: "overlapping mutable borrow", locations).
11. Sui gotcha: Borrow checker + Sui storage invariants (shared, frozen, etc.).
12. PTB tips: --preview / dry-run to inspect borrows; GasCoin special rules.

**Examples:** Full card flow test (from book), plus ties to objects/functions files.
**Resources:** Direct links to book chapters + PTB page + papers mention.
**Footer navigation** to other move/ files + summary.
